# 실습 3. 실습 지시서(과제 명세서) 만들기 (55분)

## 왜 다시 만드나요?

기존 과제 명세는 보통 "구현하라"만 있고 세부가 없어서, 학생마다 다르게 구현하고 채점 분쟁이 생깁니다. 실험 과목의 "변수·위험요소·재현성"을 컴공 버전으로 바꿔 정리합니다.

### ✏️ 퀴즈 6. 개념 번역하기

실험 과목의 개념이 프로그래밍 과목에서는 무엇에 해당할까요? 빈칸을 채워보세요.

| 실험 과목 | 프로그래밍 과목에서는? |
|---|---|
| 변수(조작·종속·통제) | [ ① ] |
| 위험요소 ⚠️ | [ ② ] |
| 재현성 체크리스트 | [ ③ ] |

<details>
<summary>▶ 정답 보기</summary>

| 실험 과목 | 프로그래밍 과목 |
|---|---|
| 변수 | **① 입출력 명세 + 고정 조건** (입력 형식 / 출력 형식 / 언어·금지 함수·제한) |
| 위험요소 ⚠️ | **② 흔한 오류·주의점** (세그폴트, 무한루프, 메모리 누수) |
| 재현성 | **③ 실행 환경·채점 조건** (컴파일러·옵션, 테스트 케이스) |

즉 "누가 채점해도, 어느 학생이 구현해도 같은 기준"을 만드는 게 목적입니다.
</details>

## 따라 하기 (35분)

**1단계.** 배포된 **샘플 과제 안내문**(스택 괄호 검사기)을 프로젝트 지식에 올립니다.

**2단계.** 아래를 입력합니다.

```
방금 올린 과제 안내문을 아래 형식으로 다시 만들어줘. (아티팩트로)

■ 과제 개요: 목적과 핵심 개념 (2~3문장)
■ 입출력 명세: 입력 형식 / 출력 형식 / 고정 조건(언어·금지사항·제한)
■ 흔한 오류·주의점: 각 구현 단계 옆에 ⚠️ 표시로 배치
■ 구현 단계: 번호 / 할 일 / 왜 이 단계가 필요한가(1줄) / ⚠️ 주의
■ 실행 환경·채점 조건: 컴파일 명령, 채점 기준, 공개 테스트 케이스
■ 예상 결과와 흔한 버그: 정상 출력 예시 / 자주 나는 버그 3가지

★ 중요: 원문에 없는 정보는 지어내지 말고 "[교수 확인 필요]"로 비워둬.
★ 완성 코드는 절대 포함하지 마. 함수 원형(시그니처)까지만.
```

### ✏️ 퀴즈 7. 빈칸으로 남을 것 예상하기 — 오늘의 핵심 기법

프롬프트의 `★ 지어내지 말고 [교수 확인 필요]로 비워둬` 조건 때문에, 샘플 안내문에 없던 정보들이 빈칸으로 남게 됩니다. **어떤 항목들이 빈칸으로 남을까요?** 3개만 예상해보세요.

내 예상: [ &nbsp;&nbsp;&nbsp;&nbsp; ], [ &nbsp;&nbsp;&nbsp;&nbsp; ], [ &nbsp;&nbsp;&nbsp;&nbsp; ]

<details>
<summary>▶ 정답 보기</summary>

Claude 결과물에서 실제로 빈칸으로 남는 대표 항목들:

- `[교수 확인 필요: 최대 입력 길이]` — 원문에 입력 제한이 없었음
- `[교수 확인 필요: strlen 등 표준 함수 허용 여부]` — "스택은 직접 구현"만 있고 나머지가 애매했음
- `[교수 확인 필요: 컴파일 경고 시 감점 폭]`, `[비공개 테스트 개수·배점]` — 채점 세부가 없었음
- `[교수 확인 필요: INVALID 위치를 0부터 셀지 1부터 셀지]`

**공통점이 보이시나요?** 전부 **교수만 정할 수 있는 것들**입니다. Claude가 이걸 멋대로 지어냈다면 채점 때 사고가 났겠죠. 그래서 이 기법이 오늘 가장 중요합니다: **AI에게 빈칸을 만들라고 시키고, 그 빈칸을 내가 채운다.**
</details>

**3단계.** 만든 지시서를 검토시킵니다.

```
방금 만든 과제 명세서를 이 과제를 처음 보는 2학년 입장에서 봐줘.
- 학생마다 다르게 해석할 수 있는 모호한 문장 3개를 찾아 고쳐줘.
- 채점 시 분쟁이 생길 만한 조건(허용/금지가 애매한 것)을 짚어줘.
```

**4단계.** `[교수 확인 필요]` 빈칸을 **내가 직접** 채웁니다. (예: 최대 길이 1000, strlen 허용, 경고당 -0.5점…)

## 짝 점검 (20분)

옆자리 교수님과 화면을 바꿔 확인:
- [ ] 입출력 명세만 보고 누구나 같은 프로그램을 떠올릴 수 있는가
- [ ] ⚠️ 주의점이 해당 단계 옆에 붙어 있는가
- [ ] `[교수 확인 필요]`를 다 채웠는가
- [ ] 완성 코드가 섞여 들어가지 않았는가

# 실습 4. 학생용 사전학습 자료 만들기 (35분)

## 4-1. 가상 리허설 시나리오 (15분)

학생이 수업 전에 **게임처럼 개념을 체험**하는 자료입니다. 컴공에서는 "디버깅 상황 판단"이 핵심입니다.

```
방금 만든 과제 명세서를 바탕으로, 학생이 수업 전에 풀어볼
"가상 리허설 시나리오"를 아티팩트로 만들어줘.

- 총 5개 장면
- 각 장면: 상황 설명 → 3지선다("다음에 뭘 해야 할까?") → 선택별 결과 설명
- 한 장면은 꼭 '프로그램이 죽는(오류) 상황의 원인 추리'로
- 한 장면은 꼭 '결과가 예상과 다르게 나왔을 때' 상황으로
- 마지막에 "실습 전 자가점검 5문항"

말투: 2학년에게 친근하되 정확하게.
```

### ✏️ 퀴즈 8. 학생이 되어 풀어보기

Claude가 만들 시나리오의 한 장면입니다. **교수님이 학생이 되어** 직접 풀어보세요.

> **장면.** 괄호 검사기에 `"(("`을 넣었더니 아무 출력 없이 프로그램이 죽습니다(세그폴트). 코드에서 닫는 괄호를 만나면 `pop()`을 호출합니다. 무엇부터 의심해야 할까요?
>
> ① 컴퓨터가 느려서 그렇다 — 다시 실행해본다
> ② 빈 스택에서 pop을 시도했는지 확인한다
> ③ printf를 100개 넣어서 다 찍어본다

내 답: ( &nbsp;&nbsp; )

<details>
<summary>▶ 정답 보기</summary>

**② 방향이 맞지만, 함정이 하나 더 있습니다.**

- ①: 세그폴트는 속도 문제가 아니라 잘못된 메모리 접근. 다시 실행해도 같습니다.
- ② ✅: 빈 스택 pop은 세그폴트의 단골 원인. 그런데 잠깐 — 입력 `"(("`엔 닫는 괄호가 없으니 pop이 호출되지 않았을 텐데도 죽었다면? 그럼 **초기화 안 된 포인터** 같은 다른 원인을 의심해야 합니다. "어디서 죽는가"를 좁히는 습관이 핵심.
- ③: printf 디버깅도 방법이지만 100개는 과합니다. 위치를 이분 탐색하듯 좁혀가는 게 효율적.

이런 "한 겹 더 생각하게 하는" 장면이 좋은 리허설입니다. Claude 결과물에 이런 깊이가 없으면 "함정을 한 겹 더 넣어줘"라고 요청하세요.
</details>

## 4-2. 연습 데이터 세트 — 알고리즘 추적 연습 (15분)

```
괄호 검사기(스택) 주제로, 학생이 실습 전에 풀어볼 연습 자료를 만들어줘.

1. 추적 문제: 입력 "{[()]}"를 처리할 때 한 글자씩 스택 상태가
   어떻게 변하는지 표로 채우게 하는 문제
2. 복잡도 문제: 이 알고리즘의 시간복잡도를 구하고 이유를 쓰게 하는 문제
3. 버그 찾기: push/pop이 포함된 15줄 내외 C 코드에 버그를 하나 몰래 넣고
   찾게 하는 문제
4. 각 문제의 모범답안과 채점 유의점

★ 복잡도와 실행 결과는 내가 검증할 거야. 중간 단계를 다 보여줘.
```

### ✏️ 퀴즈 9. 추적 표 직접 채우기 (검산 연습)

Claude의 모범답안을 **교수님이 검산**해야 합니다. 연습 삼아 직접 채워보세요. 입력 `{[()]}`:

| 읽은 문자 | 동작 | 스택 상태 (왼쪽이 바닥) |
|---|---|---|
| `{` | push | `{` |
| `[` | push | `{ [` |
| `(` | push | [ ① ] |
| `)` | [ ② ] | `{ [` |
| `]` | pop, `[`와 짝 확인 ✓ | [ ③ ] |
| `}` | pop, `{`와 짝 확인 ✓ | (비어 있음) |

최종 판정: [ ④ ]

<details>
<summary>▶ 정답 보기</summary>

- ① `{ [ (`
- ② `pop, '('와 짝 확인 ✓`
- ③ `{`
- ④ 입력 끝 + 스택 비어 있음 → **VALID**

이 표를 직접 채워보신 것이 바로 **검산**입니다. 학생에게 배포하기 전에 이렇게 한 번 손으로 따라가 보는 습관 — 그게 오늘 계속 강조한 검증 원칙입니다.

⚠️ 특히 **버그 찾기 문제**는 Claude가 심은 버그가 **의도한 딱 하나뿐인지** (다른 버그가 우연히 더 들어있지 않은지) 코드를 직접 읽고 확인하세요.
</details>

## ✏️ 마무리 퀴즈. 오늘 배운 것 채우기

빈칸을 채우며 하루를 정리해보세요.

1. **[ ① ] + 지식 + 지침** = 내 과목을 기억하는 조교 → 내일·모레도 그대로 사용
2. 컴공 지침엔 "[ ② ]를 통째로 주지 마"를 꼭 넣는다
3. 아티팩트는 한 번에 쓰지 말고 "[ ③ ]"라고 말하며 다듬는다
4. 코드 동작·복잡도·계산은 반드시 [ ④ ]가 검증한다
5. 좋은 명세서의 비결은 AI에게 "[ ⑤ ]"라는 빈칸을 만들게 하는 것

<details>
<summary>▶ 정답 보기</summary>

1. ① **프로젝트**
2. ② **완성 코드**
3. ③ **"여기(만) 고쳐줘"** — 부분 수정 요청
4. ④ **교수(나)** — Claude는 자신 있게 틀릴 수 있다
5. ⑤ **[교수 확인 필요]** — 교수만 정할 수 있는 항목이 빈칸으로 남는지 확인

5개 다 맞히셨다면 오늘 완벽하게 소화하신 겁니다. 내일은 이 프로젝트에서 평가 문항과 채점 루브릭을 만듭니다. **프로젝트 지우지 마세요!**
</details>
