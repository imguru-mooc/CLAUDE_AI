## "바이브코딩"이 뭔가요? (2분)

**코드를 직접 쓰지 않고**, 말로 "이런 프로그램 만들어줘"라고 하면 Claude가 만들어주는 방식입니다. 마음에 안 들면 "여기 고쳐줘"라고 말로만 고칩니다. 이틀간 문서에 하던 것을 오늘은 **프로그램**에 하는 것뿐입니다. **코딩 경험이 없어도 됩니다.**

# 실습 1. 내 수업 AI 전략 세우기 (40분)

## 먼저 알아둘 것: 학생 AI 사용의 신호등 규칙

요즘 학생들은 다 AI를 씁니다. 막을 수 없다면 **규칙을 정합니다.**

- 🟢 **초록불**: 마음껏 사용 OK
- 🟡 **노란불**: 쓰되, 어디에 썼는지 표시할 것
- 🔴 **빨간불**: 이 단계에선 사용 금지

### ✏️ 퀴즈 1. 신호등 판단 연습

자료구조 과목의 다음 상황, 여러분이라면 무슨 색을 켜시겠어요? (정답이 하나로 정해진 문제가 아닙니다 — 먼저 **내 판단**을 적는 게 목적)

| (C) 중간고사 (지필, 시험장) | 🟢/🟡/🔴 [ &nbsp; ] |
| (D) 팀 프로젝트 보고서의 문장 다듬기 | 🟢/🟡/🔴 [ &nbsp; ] |

<summary>▶ 정답 보기 (하나의 예시 — 정답은 교수마다 다릅니다)</summary>

흔한 설정 예시: (A) 🔴 또는 🟡(설계는 AI 상담 가능, 코드는 직접) — 과제의 목적이 구현 능력이므로. (B) 🟢 — 학습 단계의 질문은 권장. (C) 🔴 — 당연. (D) 🟡 — 허용하되 "AI로 다듬은 문단 표시".

핵심은 색이 아니라 **미리, 명확하게, 모두에게 같게** 정해서 강의계획서에 쓰는 것입니다. "이거 AI로 했어요?" 분쟁의 대부분은 규칙이 없어서 생깁니다.
</details>

# 실습 2. 만들 도구 정하기 (30분)

오후 바이브코딩의 **설계도**를 그리는 시간입니다. 무작정 "만들어줘" 하면 산으로 갑니다.

**⌨️ 입력 ① — 반복연습 지점 찾기.**
```
우리 과목에서 학생들이 "머리로는 아는데 손이 안 따라오는" 영역,
즉 반복 연습이 필요한 것을 찾고 싶어. 이틀간의 자료(오개념 세트,
자주 하는 실수, FAQ)를 근거로 후보 5개를 뽑아줘.
```

> 👀 **확인**: 이틀간 쌓은 자료(재귀 오개념, 포인터 실수 등)가 근거로 쓰이는가. 후보에 "스택 상태 추적", "재귀 호출 순서" 같은 것이 보이면 정상입니다.

**⌨️ 입력 ② — 하나 고르기.**
```
그중 "웹에서 클릭·조작하며 연습하는 인터랙티브 도구"로 만들었을 때
효과가 가장 큰 것 1개를 추천해줘. 이유도 함께.
```

> 👀 **확인**: 추천에 동의하는가? 아니면 → `"나는 스택 추적이 더 급해. 그걸로 할게"` — 오늘도 결정은 내가.

**✍️ 기획서 직접 작성 (10분)** — 이번엔 입력이 아니라 **손으로** 씁니다. 이 기획서의 구체성이 오후의 품질을 결정합니다.

```
[도구 기획서 — 종이나 메모장에 직접 작성]
■ 도구 이름: (예: 스택 괄호 검사 시각화 연습기)
■ 무엇을 연습하나: (딱 한 가지만! 예: 괄호 입력에 따른 스택 push/pop 동작)
■ 언제 쓰나: 수업 전 / 수업 중 / 복습 중 (택1)
■ 핵심 동작: 학생이 (무엇)을 하면 → (무엇)이 바뀐다
  (예: 괄호 문자열을 입력하고 "한 단계" 버튼을 누르면
   → 스택 그림이 한 글자만큼 push/pop 되며 움직인다)
■ 연습 흐름: 문제 → 학생이 다음 스택 상태를 예측 입력 → 즉시 채점 → 해설 → 다음
■ 화면에 꼭 있을 것 3가지: (예: 스택 그림, 현재 읽는 문자 강조, 정오 카운트)
■ 넣지 말 것: 로그인, 회원가입, 서버 저장 (단순하게!)
```

어느 기획이 오후에 좋은 도구가 될까요?

- (나) 학생이 괄호 문자열을 입력하고 "한 단계"를 누르면 → 스택 그림이 push/pop 되며 한 칸씩 움직인다

내 답: ( &nbsp;&nbsp; ) — 이유: [ &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; ]

**(나)**. 사흘 내내 나온 그 기준 — **관찰 가능·구체적**. (가)는 목표지 동작이 아닙니다. Claude는 (나)처럼 "무엇을 누르면 무엇이 어떻게 변한다"가 있어야 정확히 만듭니다. 학습목표(1일차), 루브릭 기술어(2일차), 도구 기획(3일차) — 전부 같은 원리입니다.

# 실습 3. 바이브코딩: 도구 만들기 (95분) ⭐ 오늘의 하이라이트

## 3-1. 첫 생성 (15분)

**⌨️ 입력 ①** — 기획서는 지침처럼 **통째로 붙여넣는 예외**입니다. (설계도니까)
```
아래 기획서대로 학생용 연습 도구를 아티팩트(HTML 한 파일)로 만들어줘.

"""
(방금 손으로 쓴 기획서 전체 붙여넣기)
"""

조건:
- 브라우저에서 바로 열리게 (설치·서버 없이)
- 글씨 크게, 밝은 배경, 휴대폰에서도 되게
- 판정 로직(push/pop 규칙)은 코드 주석에 설명을 써줘 (내가 검증하게)
```

> 👀 **확인**: 오른쪽에 **실제로 조작되는** 도구가 떴는가. 버튼을 눌러보세요. 안 뜨거나 오류 → `"이 오류를 고쳐줘"` + 화면의 오류 메시지 붙여넣기.

## 3-2. 고쳐 나가기 — 대화 수정 루프 (40분)

첫 결과물은 완벽하지 않습니다. **직접 학생처럼 써보고**, 발견한 것을 말로 고칩니다. 이틀간 문서에 하던 것과 똑같습니다.

### ✏️ 퀴즈 4. 좋은 수정 요청 고르기

도구를 써보니 스택 그림이 작아서 잘 안 보입니다. 어느 요청이 잘 고쳐질까요?

- (가) 좀 더 좋게 만들어줘
- (나) 스택 그림을 화면 절반 크기로 키우고, 방금 push된 칸은 색을 다르게 강조해줘

내 답: ( &nbsp;&nbsp; )

<details>
<summary>▶ 정답 보기</summary>

**(나)**. (가)는 Claude가 "좋게"의 방향을 몰라 엉뚱한 곳을 고칩니다. 수정 요청도 기획서와 같은 원리 — **무엇을 어떻게**. 아래 상황별 문장들을 참고해 최소 3회 이상 수정 루프를 돌려보세요.
</details>

**상황별 수정 입력 (필요한 것을 골라 사용, 본인 말로 바꿔도 됨):**

**⌨️ 수정 입력 A — 보이는 문제**
```
스택 그림을 화면 절반 크기로 키우고, 방금 push된 칸은 색으로 강조해줘.
현재 읽고 있는 문자도 문자열에서 밑줄로 표시해줘.
```

**⌨️ 수정 입력 B — 교육적 개선**
```
학생이 다음 상태를 틀리게 예측하면 정답을 바로 보여주지 말고,
힌트("지금 읽은 문자는 닫는 괄호야. 스택에서 뭘 해야 할까?")를 먼저 주고
두 번 틀리면 그때 정답과 해설을 보여주게 바꿔줘.
```

**⌨️ 수정 입력 C — 극단 입력 방어**
```
빈 문자열, 괄호가 아닌 문자, 아주 긴 입력을 넣어봤더니 (발견한 문제).
이런 극단 입력에서도 오류 없이 안내 문구가 나오게 고쳐줘.
```

**⌨️ 수정 입력 D — 난이도 구조**
```
3단계 레벨을 넣어줘. 레벨 1은 소괄호만, 레벨 2는 3종 괄호,
레벨 3은 교차·중첩이 섞인 문자열. 각 레벨 5문제 연속 정답 시 다음 레벨.
```

> 👀 **매 수정 후 확인**: 요청한 곳만 바뀌었는가, 기존 기능이 깨지지 않았는가. 깨졌으면 → `"레벨 기능을 넣은 뒤로 '한 단계' 버튼이 안 눌려. 고쳐줘"`

## 3-3. 로직 검증 — 사흘째의 그 원칙 (15분)

내 답: [ &nbsp;&nbsp;&nbsp; ], [ &nbsp;&nbsp;&nbsp; ], [ &nbsp;&nbsp;&nbsp; ]


> 👀 **확인**: 고친 후 4개 케이스가 전부 맞는가. **직접 다시 넣어서** 확인 — Claude의 "고쳤다"는 말이 아니라 화면으로.

### ✏️ 퀴즈 7. 맡길 일 / 안 맡길 일

3일간의 경험을 바탕으로 분류해보세요.

| 업무 | AI에 맡긴다 / 안 맡긴다 |
|---|---|
| (A) 기출 변형 문항 초안 생성 | [ &nbsp;&nbsp;&nbsp; ] |
| (B) 계산 문항의 최종 정답 확정 | [ &nbsp;&nbsp;&nbsp; ] |
| (C) 실제 학생 답안을 올려 서술형 채점 | [ &nbsp;&nbsp;&nbsp; ] |
| (D) FAQ·피드백 문구 초안 | [ &nbsp;&nbsp;&nbsp; ] |
| (E) 최종 성적(등급) 판단 | [ &nbsp;&nbsp;&nbsp; ] |

<details>
<summary>▶ 정답 보기</summary>

(A) 맡긴다(검토는 내가) (B) **안 맡긴다** — 검산은 교수 (C) **안 맡긴다** — 학생 개인정보 + 평가 책임 (D) 맡긴다(톤·정확성 검토는 내가) (E) **안 맡긴다** — 평가의 최종 책임.

패턴이 보이죠: **초안은 AI, 검증·개인정보·최종 판단은 교수.** 이게 프로토콜의 뼈대입니다.
</details>

## 따라 하기 — 입력 3개

**⌨️ 입력 ① — 뼈대부터. (사흘째 같은 패턴)**

```
4. 윤리·학술 정직성  5. 개인정보·데이터 보호
각 섹션에 이 프로젝트의 3일치 대화에서 나온 실제 결정·원칙을 반영해줘.
```

> 👀 **확인**: 3일간의 실제 결정들("⚠️ 교수 검증 필요", 완성 코드 금지, 신호등 확정표, 검산 원칙, 기출만 업로드…)이 뼈대에 나타나는가. 일반론뿐이면 → `"우리가 이 프로젝트에서 실제로 정한 규칙들을 근거로 다시 뽑아줘"`

**⌨️ 입력 ② — 본문 작성.**
```

- 문체: 강의계획서 첨부용, 공식적이되 간결하게
- 3번 섹션에는 오전에 확정한 신호등 표와 AI 사용 기록 양식 포함
- 마지막에 "본 프로토콜은 매 학기 시작 전 재검토한다" 개정 조항
```

**⌨️ 입력 ③ — 학생 눈으로 검토.**
```

"뭐가 되고 뭐가 안 되는지" 헷갈리는 부분이 있으면 지적하고 고쳐줘.
```

### ✏️ 퀴즈 8. 윤리 6원칙 — 빈칸으로 최종 점검

프로토콜에 꼭 들어가야 할 6가지. 빈칸을 채우세요.

6. 학생 AI 규칙은 전원 동일 적용, 강의계획서에 [ ⑥ ] 고지한다
