## 먼저 알아둘 것: 평가 문항의 4가지 유형 (5분)

| 유형 | 무엇을 보나 | 자료구조 과목 예시 |
|---|---|---|
| **계산형** | 절차·공식 적용의 정확성 | "이 스택 연산 후 상태를 쓰시오" |
| **개념형** | 원리 이해·오해 판별 | "배열 스택과 연결 스택의 차이를 서술하시오" |
| **응용형** | 실제 상황에 전이 | "이 로그 데이터에 적합한 자료구조를 고르고 근거를 대시오" |
| **설계형** | 종합·창안 | "메모리 제한 하에서 자료구조를 설계하시오" |

### ✏️ 퀴즈 1. 유형 분류 연습

다음 문항은 각각 어떤 유형일까요?

| 문항 | 유형 |
|---|---|
| (A) "다음 함수의 시간복잡도를 Big-O로 구하시오" | [ &nbsp;&nbsp;&nbsp; ] |
| (B) "편의점 대기줄 관리 프로그램에 스택이 아니라 큐를 써야 하는 이유를 실제 상황에 근거해 설명하시오" | [ &nbsp;&nbsp;&nbsp; ] |
| (C) "검색은 빠르되 메모리는 1MB 이하인 학생 명부 자료구조를 설계하고 트레이드오프를 정당화하시오" | [ &nbsp;&nbsp;&nbsp; ] |

<details>
<summary>▶ 정답 보기</summary>

(A) **계산형** — 정해진 절차(복잡도 분석)를 적용해 하나의 답을 냄
(B) **응용형** — 개념을 실제 맥락(대기줄)으로 전이. FIFO라는 개념만 묻는 게 아니라 상황 판단을 요구
(C) **설계형** — 제약조건 안에서 해를 만들고 정당화. 정답이 여러 개 → **루브릭이 필요** (오후 실습 3)

Claude와의 궁합도 유형마다 다릅니다: 계산형은 **변형 대량 생산**에 강하지만 **수치를 자신 있게 틀리고**, 개념형은 **매력적 오답 설계**가 최강, 설계형은 **제약조건 명세화**를 잘합니다.
</details>

# 실습 1. 내 시험 진단하기 (35분)

**0단계.** 배포된 **샘플 기출문항**(지난 학기 중간고사)을 어제 프로젝트의 지식에 올립니다.

> ⚠️ **이번 학기에 낼 문제는 절대 올리지 않습니다.** 반드시 **지난 학기 기출**만. (출제 보안)

### ✏️ 퀴즈 2. 넣기 전에 예측!

샘플 기출(8문항, 100점)을 훑어보고 예상해보세요. 계산형(추적·구현 포함) 비율은 몇 % 정도일까요? 응용형·설계형은 몇 문항일까요?

내 예상: 계산형 약 [ &nbsp;&nbsp; ]% / 응용형 [ &nbsp;&nbsp; ]문항 / 설계형 [ &nbsp;&nbsp; ]문항

<details>
<summary>▶ 정답 보기</summary>

배점 기준 계산·추적·구현형이 **약 85%**(문제 1~7), 개념형은 문제 8 하나(15%), **응용형·설계형은 0문항**입니다.
일부러 이렇게 만든 샘플입니다 — 자연이공 계열 시험의 전형적인 편중이거든요. 이제 Claude에게 진단시키고, 여러분 눈과 비교해봅니다.
</details>

**⌨️ 입력 ①** — 분류부터.
```
방금 올린 기출문항의 각 문항을 계산형/개념형/응용형/설계형으로
분류하고, 분류 근거를 한 줄씩 표로 정리해줘.
```

> 👀 **확인**: 8문항이 전부 분류됐는가. 분류가 내 생각과 다른 문항이 있으면 → `"6번은 변환 절차를 따라 하는 거니까 계산형 아니야? 네 근거를 다시 설명해줘"` — 분류 기준을 **토론**하는 것도 훈련입니다.

**⌨️ 입력 ②** — 표에 집계를 쌓습니다.
```
좋아. 유형별 비율을 두 가지 기준으로 집계해줘.
문항 수 기준과 배점 기준. 간단한 표로.
```

> 👀 **확인**: 문항 수 기준과 **배점 기준**이 다르게 나오는가. (배점 기준이 실제 성적에 미치는 영향입니다)

**⌨️ 입력 ③** — 어제 자료와 연결합니다.
```
어제 정리한 학습목표 표와 이 시험을 대조해줘.
평가에서 확인되지 않는 학습목표가 있으면 짚어줘.
```

> 👀 **확인**: "설계 축 목표(자료구조 선택·정당화)가 시험에 없다" 같은 지적이 나오는가. — 어제 만든 자료가 오늘 근거로 쓰이는 순간입니다. 프로젝트의 힘이죠.

**⌨️ 입력 ④** — 처방까지. 단, 아직 만들지는 못하게.
```
이 시험의 균형을 개선하려면 어떤 유형의 문항을 추가해야 할지,
어떤 학습목표를 겨냥해야 할지 제안해줘. 문항 자체는 아직 만들지 마.
```

> 👀 **확인**: "응용형·설계형 추가, 3번 목표(자료구조 선택) 겨냥" 방향의 제안인가. — 이 제안이 실습 2의 **주문서**가 됩니다.

# 실습 2. 새 문항 만들기 — 3단 리듬 (65분)

## 왜 3단인가?

"문제 만들어줘" 한 방은 그럴듯하지만 엉성한 문제를 낳습니다. 오늘의 리듬: **① 구조 분석 → ② 생성 → ③ 자체 검토.**

## 2-1. 응용형 문항 세트 (30분)

**⌨️ 입력 ① — 시드 분석부터. (만들지 말고)**
```
기출 8번(배열 스택 vs 연결 스택 비교)을 시드로 응용형 문항을 만들려고 해.
지금은 만들지 말고 8번의 구조부터 분석해줘: 측정하는 역량 / 필요한
선수지식 / 응용형으로 바꾸려면 무엇을 더해야 하는지.
```

> 👀 **확인**: "실제 상황(맥락)을 더해야 한다"는 방향이 나오는가. 분석이 얕으면 → `"우리 학생들이 접해본 상황(게임, SNS, 편의점 알바)을 맥락 후보로 3개 제안해줘"`

**⌨️ 입력 ② — 쉬움 1문항만 먼저.**
```
좋아, 그 방향으로 응용형 문항을 만들자. 먼저 난이도 "쉬움" 1문항만:
문제 / 모범답안 / 단계별 풀이 / 배점과 부분점수 기준.
계산이 있으면 모든 중간값을 표기해줘.
```

> 👀 **확인**: 부분점수 기준이 **채점 가능한 형태**인가("근거 1개당 3점" 등). 여기서 형식을 확정해야 나머지 문항이 같은 품질로 나옵니다. 마음에 안 들면 지금 고치세요.

**⌨️ 입력 ③ — 형식 확정 후 나머지.**
```
이 형식 좋아. 같은 형식으로 "보통" 2문항과 "어려움" 1문항을 추가해서
문항 세트를 아티팩트로 정리해줘. 어려움 문항은 두 개 이상의
자료구조를 비교·선택하게 해줘.
```

**⌨️ 입력 ④ — 자체 검토. (오늘의 마법)**
```
방금 만든 4문항을 출제 검토위원 입장에서 스스로 점검해줘.
- 계산이 있으면 다른 방법으로 재계산해 답 일치 확인
- 조건이 애매하거나 복수 해석이 가능한 부분 지적
- 학생이 이의 제기할 만한 지점 예측과 수정안
```

> 👀 **확인**: Claude가 실제로 자기 문항의 약점을 찾아내는가. 지적이 나오면 → `"2번 지적을 반영해서 문항을 수정해줘"`

### ✏️ 퀴즈 3. 그런데 이걸로 끝일까요?

입력 ④의 자체 검토를 통과했습니다. 이 문항 세트를 바로 시험에 낼 수 있을까요? (O/X, 이유 한 줄)

내 답: ( &nbsp;&nbsp; ) — 이유: [ &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; ]

<details>
<summary>▶ 정답 보기</summary>

**X**. AI의 자체 검토는 **1차 거름망**일 뿐입니다. 특히 계산이 포함된 문항의 **최종 검산은 교수가 직접** 합니다 — Claude는 "재계산했고 일치한다"고 말하면서 **둘 다 틀릴 수** 있습니다. 어제 퀴즈 9에서 손으로 스택을 추적했던 것처럼, 배포·출제 전 30초 검산이 원칙입니다.
</details>

## 2-2. 오개념 함정 문제 (35분) — 오늘의 하이라이트

Claude가 가장 잘하는 것: 학생의 **오해를 겨냥한 정밀 함정(매력적 오답)** 설계.

### ✏️ 퀴즈 4. 넣기 전에 예측!

주제를 "재귀(recursion)"로 잡겠습니다. 2학년이 재귀에서 흔히 하는 오해, 두 개만 먼저 예상해보세요.

내 예상 1: [ &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; ] / 내 예상 2: [ &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; ]

<details>
<summary>▶ 정답 보기 (Claude의 답 예시)</summary>

1. "재귀는 반복문보다 **항상** 느리다/메모리를 많이 쓴다" (일반화 오류)
2. "기저 조건(base case)이 없어도 언젠가 알아서 멈춘다"
3. "재귀 호출이 **동시에** 실행된다" (호출 스택 개념 부재)
4. "return 문의 재귀 호출 결과가 자동으로 누적된다"
5. "꼬리 재귀와 일반 재귀는 완전히 같은 것이다"

겹친 게 있나요? 내 예상에만 있는 오해가 있다면 그것도 다음 입력에서 추가합니다.
</details>

**⌨️ 입력 ① — 오해 목록부터.**
```
"재귀" 주제로 오개념 함정 문제를 만들려고 해. 먼저 우리 2학년이
흔히 하는 오해 5개를 나열하고, 왜 그렇게 오해하게 되는지
원인도 한 줄씩 설명해줘. 아직 문제는 만들지 마.
```

> 👀 **확인**: 내 예상과 대조. 내 예상에만 있는 오해가 있으면 → `"우리 학생들은 ○○라는 오해도 많아. 6번으로 추가해줘"`

**⌨️ 입력 ② — 1문항만 먼저, 구조 확인.**
```
오해 1번과 3번을 오답 선택지로 쓰는 4지선다 1문항을 먼저 만들어줘.
각 오답 옆에 (어떤 오해를 겨냥했는지) 주석을 달고,
정답 해설과 각 오답을 고른 학생용 피드백 한 줄도 포함해줘.
```

### ✏️ 퀴즈 5. 매력적 오답 판별하기

Claude가 만든 문항을 보기 전에, 감을 잡아봅시다. "재귀 함수 f(3)의 호출 순서"를 묻는 문항에서 **좋은 오답**은 어느 쪽일까요?

- (가) `f(3) → f(5) → f(7)` (아무렇게나 커지는 순서)
- (나) `f(3), f(2), f(1)이 동시에 실행된 후 한꺼번에 반환` (오해 3번을 정확히 반영)

내 답: ( &nbsp;&nbsp; )

<details>
<summary>▶ 정답 보기</summary>

**(나)**. (가)는 아무도 안 고르는 "버리는 오답"이라 4지선다가 사실상 3지선다가 됩니다. (나)는 "동시 실행" 오해를 가진 학생만 정확히 걸리는 **정밀 함정**이고, 학생이 (나)를 골랐다면 어떤 오해를 하고 있는지 **진단**까지 됩니다. 오답에 주석을 달게 하는 이유가 이것입니다.
</details>

> 👀 **확인 (입력 ② 결과)**: 오답들이 (나)처럼 특정 오해를 정밀하게 반영하는가. 버리는 오답이 보이면 → `"3번 선택지는 아무도 안 고를 것 같아. 오해 5번을 반영한 오답으로 바꿔줘"`

**⌨️ 입력 ③ — 나머지 문항.**
```
좋아. 같은 형식으로 2문항을 더 만들어서, 총 3문항을
"재귀 오개념 진단 세트" 아티팩트로 정리해줘.
남은 오해들(2, 4, 5번)도 골고루 오답에 배치해줘.
```

**⌨️ 입력 ④ — 오후를 위한 저장.**
```
방금 만든 오답별 피드백 문구들만 따로 모아서
"오답 피드백 모음" 표로 정리해줘. (오후에 피드백 뱅크로 확장할 거야)
```

> 👀 **확인**: 피드백이 "틀렸습니다"가 아니라 **오해를 교정하는 방향**("재귀 호출은 동시에가 아니라 스택에 쌓이며 하나씩…")인가.

# 실습 3. 채점 루브릭 만들기 + 시운전 (65분)

## 왜 필요한가요?

정답이 열려 있는 과제(구현 과제·보고서·프로젝트)는 루브릭 없이는 채점자마다 점수가 달라져 분쟁이 생깁니다. 오늘은 **어제 만든 과제 명세서(스택 괄호 검사기)**의 채점 루브릭을 만듭니다 — 어제 자료의 재활용입니다.

### ✏️ 퀴즈 6. 좋은 기술어 고르기

루브릭의 "코드 품질" 기준, 어느 기술어가 좋을까요?

- (가) 코드가 깔끔하고 성실하게 작성되었다
- (나) 함수가 단일 기능으로 분리되어 있고, 변수명이 역할을 드러내며, 핵심 로직에 주석이 있다

내 답: ( &nbsp;&nbsp; ) — 핵심 단어: [ &nbsp;&nbsp;&nbsp;&nbsp; ]

<details>
<summary>▶ 정답 보기</summary>

**(나)**. 핵심 단어는 어제 배운 그 단어 — **"관찰 가능"**. (가)의 '깔끔·성실'은 채점자마다 다르게 봅니다. (나)는 코드를 보면 누구나 같은 판단이 나옵니다. 참고로 (가)는 어제 배포된 샘플 과제 안내문의 채점 기준 그대로입니다 — 오늘 그걸 고치는 겁니다.
</details>

## 따라 하기 — 입력 5개 (30분)

**⌨️ 입력 ① — 기준 축부터. (아직 표는 만들지 마)**
```
어제 만든 괄호 검사기 과제 명세서의 채점 루브릭을 만들자.
먼저 평가 기준을 지식/기능/태도 3축으로 나눠서 기준 항목만
목록으로 제안해줘. 아직 수준 기술어 표는 만들지 마.
```

> 👀 **확인**: 기능 축에 "경계 케이스 처리", 태도 축에 "제출 규정 준수·기여 명시(페어)" 같은 항목이 있는가. 빠진 게 있으면 → `"태도 축에 '페어 기여 내역을 주석에 명시'도 추가해줘"` (강의계획서의 페어 제출 규정 반영)

**⌨️ 입력 ② — 표 작성.**
```
좋아, 그 기준으로 루브릭 표를 아티팩트로 만들어줘.
각 기준을 4수준(탁월-충족-부분충족-미충족)으로 나누되,
기술어는 반드시 관찰 가능한 행동·산출물 특징으로 써줘.
"깔끔하다", "성실하다" 같은 추상어는 금지.
```

> 👀 **확인**: 추상어가 정말 없는가. 하나라도 보이면 → `"'적절히 처리'는 추상적이야. 무엇을 보면 판단할 수 있는지로 다시 써줘"`

**⌨️ 입력 ③ — 배점.**
```
과제 총점 6점(강의계획서 기준) 배점을 기준별로 나눠줘.
배분 근거도 한 줄씩. 채점자마다 판단이 갈리기 쉬운 기준이 있으면
표시하고, 판단 예시(anchor) 문장을 붙여줘.
```

**⌨️ 입력 ④ — 시운전 준비: 가상 학생 3명.**
```
이 루브릭을 시운전할 거야. 가상 학생 제출물 요약 3개를 만들어줘.
- 학생 A: 동작은 완벽한데 코드가 한 함수에 다 들어있고 주석 없음
- 학생 B: 구조는 훌륭한데 "([)]" 같은 교차 케이스에서 오답
- 학생 C: 코드가 인터넷 예제와 거의 동일하고 변수명만 바꾼 흔적
```

### ✏️ 퀴즈 7. 시운전 전에 예측 — 구멍 찾기

방금 만든 내 루브릭으로 학생 C(표절 의심)를 채점하면 몇 점이 나올 것 같나요? 6점 만점에:

내 예상: [ &nbsp;&nbsp; ]점 — 이 점수가 적절한가요? ( 예 / 아니오 )

<details>
<summary>▶ 정답 보기</summary>

많은 경우 학생 C가 **의외로 높은 점수**(4~5점)를 받습니다. 동작·구조 기준은 다 충족하니까요. 표절 의심을 잡는 기준이 루브릭에 없다면 — 그게 바로 **구멍**입니다. 실제 학생으로 이걸 겪으면 학기가 곤란해지지만, 가상 학생으로 미리 돌리면 배포 전에 고칠 수 있습니다. 다음 입력에서 확인합니다.
</details>

**⌨️ 입력 ⑤ — 시운전 실행.**
```
학생 A, B, C를 방금 만든 루브릭으로 각각 채점해서 점수를 산출해줘.
채점 결과가 직관과 어긋나는 학생이 있으면, 루브릭의 어느 기준을
어떻게 고쳐야 하는지 제안해줘.
```

> 👀 **확인**: 학생 C의 점수가 직관과 맞는가? 안 맞으면 → `"C가 4점인 건 이상해. '독자적 구현 여부' 기준을 추가하고 강의계획서의 표절 0점 규정과 연결해줘"` → 수정된 루브릭으로 **C를 재채점**시켜 구멍이 막혔는지 확인.

## 짝 채점 실험 (20분)

옆자리 교수님과 루브릭(아티팩트)을 교환하고, **서로의 가상 학생 B**를 상대 루브릭으로 채점해보세요.

- [ ] 상대 루브릭만 보고 채점한 점수가 상대의 채점과 **±0.5점 이내**로 수렴하는가
- [ ] 수렴하지 않았다면 어느 기술어가 모호했는가 → 그 자리에서 수정
- [ ] 태도 축이 "인상"이 아니라 관찰 가능한 증거로 서술됐는가

# 실습 4. FAQ와 피드백 뱅크 만들기 (35분)

## 4-1. 오개념 FAQ — 입력 3개 (18분)

매 학기 반복되는 학생 질문을 미리 정리해 LMS에 올려두는 자료입니다.

**⌨️ 입력 ① — 질문 목록부터. (본문은 나중)**
```
"재귀와 정렬" 단원(9~10주차)의 학생용 FAQ를 만들자.
먼저 학생들이 자주 묻는 질문 8개만 한 줄씩 목록으로 제안해줘.
아직 답변은 쓰지 마.
```

> 👀 **확인**: 질문이 실제 학생 말투("퀵 정렬이 왜 퀵이에요? 느릴 때도 있다면서요")에 가까운가. 너무 교과서적이면 → `"학생들이 실제로 묻는 말투로 바꿔줘"` / 내가 매 학기 받는 질문이 빠졌으면 → `"'재귀 문제는 어떻게 시작해야 할지 모르겠어요'도 추가해줘"`

**⌨️ 입력 ② — 확정 후 본문.**
```
좋아, 그 8개로 확정. 각 질문에 [짧은 답(3줄) + 자세한 설명]을 달아서
FAQ 아티팩트를 만들어줘. 난이도 ★~★★★도 표시해줘.
말투: 시험 전날 읽는 학생 기준, 친근하되 정확하게.
```

**⌨️ 입력 ③ — 오전 자료와 합치기.**
```
오전에 만든 "재귀 오개념 진단 세트"의 오해 5가지를
"흔한 오해 교정" 섹션으로 FAQ 맨 아래에 추가해줘.
각 오해마다 [왜 틀렸나 / 올바른 이해 / 확인 미니퀴즈 1문제] 형식으로.
```

> 👀 **확인**: 오전 대화의 오해 목록이 그대로 이어지는가 — 한 프로젝트 안에서 자료가 서로 연결되는 것을 체감하는 지점입니다.

## 4-2. 피드백 뱅크 — 입력 2개 (17분)

### ✏️ 퀴즈 8. 좋은 피드백 고르기

부분충족 수준의 학생에게 줄 피드백, 어느 쪽이 좋을까요?

- (가) 경계 케이스 처리가 부족합니다. 더 성실하게 테스트하세요.
- (나) 기본 케이스는 정확히 동작합니다. 다음 제출에서는 빈 문자열과 "((("처럼 짝 없는 입력 2가지를 직접 테스트에 추가해보세요.

내 답: ( &nbsp;&nbsp; ) — (가)의 문제점 두 가지: [ &nbsp;&nbsp;&nbsp;&nbsp;&nbsp;&nbsp; ]

<details>
<summary>▶ 정답 보기</summary>

**(나)**. (가)의 문제: ① "성실하게"는 **사람(인격)을 평가**하는 말이고 ② 뭘 하라는 건지 **실행 가능한 행동이 없습니다**. 좋은 피드백 공식 = **잘한 점 확인 + 다음에 할 구체적 행동 1가지.** (나)는 학생이 오늘 밤에 바로 실행할 수 있죠.
</details>

**⌨️ 입력 ①**
```
실습 3의 루브릭에서 각 기준 × 각 수준(4수준)에 대한
학생 피드백 문구를 표로 만들어줘. 규칙:
- 각 2~3문장: ① 현재 잘한 점 ② 다음 제출에서 할 구체적 행동 1가지
- 사람이 아니라 행동·산출물에 대해서만 서술 (비난·인격 평가 금지)
- 오전에 만든 "오답 피드백 모음"의 톤과 통일해줘
```

> 👀 **확인**: "부분충족·미충족" 칸에 **오늘 밤 실행 가능한 행동**이 들어 있는가. 추상적이면 → `"미충족 칸의 행동을 더 구체적으로. 파일명·함수명 수준으로"`

**⌨️ 입력 ② — 사용법까지.**
```
이 피드백 뱅크의 맨 위에 "사용 안내" 3줄을 추가해줘:
채점 시 해당 칸을 복사하고, 학생별로 한 줄만 개인화해서 쓰라는 내용.
```

> ✅ **활용법**: 채점 때 표에서 복사 + 학생별 한 줄 개인화 → 40명 채점에도 피드백 품질 유지, 시간은 대폭 절약.

# 실습 5. 검증하고 마무리 (20분)

## 5-1. 웹 검색으로 사실 확인 — 입력 1개 (7분)

**⌨️ 입력 ①** — 응용형 문항에 실제 사례·수치를 썼다면.
```
오늘 만든 응용형 문항 세트에 포함된 사실 정보(기술 사례, 수치,
"실제로 ○○에 쓰인다" 류의 서술)를 웹 검색으로 확인해줘.
출처를 알려주고, 확인 안 되는 건 "확인 불가"로 표시해줘.
수정이 필요하면 수정안까지.
```

> 👀 **확인**: 출처가 붙는가. "확인 불가" 항목은 문항에서 빼거나 일반화하는 것이 안전합니다.

| (C) 타 대학 기출을 통째로 넣고 "거의 똑같게 만들어줘"라고 해도 된다 | [ &nbsp; ] |
